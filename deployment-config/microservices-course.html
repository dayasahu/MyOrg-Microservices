docker network create observability

make up IMAGE_TAG=v1.0.0

3

4

Creates the microservices namespace, applies Kustomize manifests, updates deployment images, then waits for all 5 rollouts to complete.

5

Applies the API Gateway ingress for api.mybank.com.

🏛️ Architecture Deep Dive

Five services, two networks, one entry point. Here's how traffic flows end to end.

Service Responsibilities

Single entry point at :8888. Validates JWT on every request (except /auth/**), strips route prefix, and proxies to the correct downstream service. Reactive (non-blocking).

Spring Cloud Gateway
👤

REST API for employee data backed by H2. Also calls the department service reactively via WebClient for the /departments endpoint.

Spring MVC + WebFlux
🏢

Lightweight REST API returning department data from a service layer. No database. Demonstrates the pure service layer pattern.

Spring MVC
🔐

Issues signed JWTs for a given userId. Exposes Swagger UI. Standalone — no configserver dependency, no database (user lookup is a pluggable stub).

Spring Security + JJWT
⚙️

Spring Cloud Config Server in native profile. Serves property files from the filesystem — mounted as a volume (Docker) or ConfigMap (K8s).

Spring Cloud Config

⚙️ Centralized Configuration

One of the most important microservices patterns: externalizing configuration so images are environment-agnostic.

💡
Core principle: Build once, run anywhere. The Docker image never changes — only the config mounted into it changes between environments.

Three-Tier Configuration Model

Profile-Based Config (Dev / Prod)

# configserver serves different files based on active profile:
config/props/employee/employee.properties       # shared / default
config/props/employee/employee-dev.properties   # active when SPRING_PROFILES_ACTIVE=dev
config/props/employee/employee-prod.properties  # active when SPRING_PROFILES_ACTIVE=prod

# Services request their config via HTTP on startup:
GET http://configserver:8071/employee/dev
# → returns merged view: default + dev overrides
⚠️
Design decision: configserver uses the native profile (filesystem), not Git. This is intentional for training simplicity. In production you'd use Spring Cloud Config's Git backend for version-controlled, auditable config changes.

🔐 Security — JWT at the Gateway

Authentication is enforced at a single chokepoint — the gateway — rather than scattered across every service.

How It Works

1

Client requests a token

POST /auth/token {"userId":"123"} → auth-service looks up user claims, signs a JWT with HMAC-SHA256, returns {"token":"eyJ..."}.

2

Client calls any API

Attaches Authorization: Bearer eyJ... to every request sent to the gateway (:8888).

3

Gateway validates locally

JwtAuthenticationFilter (order -1, highest priority) verifies the signature using the shared secret. No network call to auth-service — fast and stateless.

4

Forward or reject

Valid token → request forwarded to the upstream service. Invalid/missing token → 401 Unauthorized returned immediately, upstream never sees the request.

Design Rationale

✅ Strengths

  • 🔒Single auth enforcement point — downstream services need no auth code
  • ⚡Stateless JWT verification — no database or auth-service call per request
  • 📦Shared secret pattern is simple to understand and implement
  • 🔑Claims (roles, email, dept) travel with the token for downstream use

⚠️ Trade-offs

  • 🔑Shared secret must be kept in sync between auth-service and gateway
  • ♻️Tokens can't be revoked before expiry (no token blacklist)
  • 🔓Internal service-to-service calls (employee → department) are unauthenticated
  • 🧪UserLookupService is a stub — any userId gets a valid token

📡 Observability — The Three Pillars

Metrics, logs, and traces are all collected and correlated. Find any request's full journey in Grafana.

Trace ID in Every Log Line

The Logback pattern injects the current OTel trace context into every log message automatically:

# logback-spring.xml pattern
logging.pattern.level=%5p [${spring.application.name},%X{traceId:-},%X{spanId:-}]

# Example log output:
 INFO [employee,4bf92f3577b34da6a3ce929d0e0e4736,00f067aa0ba902b7] Returning employees: [...]
        ↑ service name    ↑ traceId (same across all services)    ↑ spanId (unique per service hop)
✅
Correlation superpower: When a request flows Client → Gateway → Employee → Department, all four log lines share the same traceId. In Grafana, search Loki for the traceId and you see all four service logs. Click the traceId and Tempo shows you the exact timing of each service hop.

⚖️ Design Decisions & Trade-offs

Every architecture choice has a cost. Here's why these decisions were made and what you give up.

🔧 Spring Cloud Config Server (native/filesystem) vs Git backend▼
Chosen: Native profile reading from mounted filesystem.

Why: Simpler setup for learning — no git repository needed for config. The Docker volume mount or K8s ConfigMap replace the git repo transparently.

Trade-off: No version history for config changes, no pull-request workflow for config updates, no rollback by git revert. In production you'd use the Git backend for auditability and rollback capability.
🔐 Stateless JWT validation at gateway vs introspection/session▼
Chosen: Gateway validates JWT locally with the shared secret. No call to auth-service per request.

Why: Zero added latency per request, no single point of failure on the auth-service for every API call, horizontally scalable gateway.

Trade-off: Tokens cannot be revoked before expiry (24h by default). A compromised token stays valid until expiry. Production mitigations: shorter expiry, token rotation, Redis-based blacklist, or switch to OAuth2/OIDC with introspection.
☸️ Kubernetes for services + Docker Compose for observability▼
Chosen: Hybrid — microservices in K8s, observability in Docker Compose.

Why: K8s teaches the real deployment patterns (ConfigMaps, Secrets, Ingress, rolling deploys). But Loki/Tempo/Prometheus/Grafana Helm charts add significant complexity for a training environment. Docker Compose runs them quickly and reliably.

Trade-off: The observability stack is not highly available and has no K8s lifecycle management. For a production cluster you'd deploy the LGTM stack to K8s using the official Helm charts or the Grafana Operator.
📡 OTel push (OTLP) vs Prometheus pull for metrics▼
Chosen: Both — metrics are pushed via OTLP to the collector AND exposed via /actuator/prometheus for Prometheus to scrape.

Why: Demonstrates both models. The OTLP push path works in K8s without needing to configure Prometheus scrape targets. The Prometheus pull endpoint is a fallback and useful for local testing with curl.

Trade-off: Slight duplication — the same metric reaches Prometheus via two paths. In production, pick one and stick to it.
🗄️ H2 in-memory database vs persistent database▼
Chosen: H2 in-memory for the employee service.

Why: Zero setup — no PostgreSQL pod needed. The learning focus is architecture patterns, not data persistence.

Trade-off: All data is lost on pod restart. Cannot run multiple replicas of employee (each would have its own empty DB). For production: swap H2 for PostgreSQL, add a PersistentVolumeClaim, and handle database migrations with Flyway or Liquibase.
⚡ Spring MVC + WebFlux together in employee service▼
Chosen: Both spring-boot-starter-web (servlet) and spring-boot-starter-webflux (reactive) in the same service.

Why: WebClient (from WebFlux) is the recommended non-blocking HTTP client for Spring — even in a servlet app. Having webflux on the classpath enables it without switching the entire app to reactive.

Trade-off: Mixed paradigms are confusing. The /departments endpoint returns Mono<String> inside a blocking MVC controller. Spring MVC handles this correctly in 3.x but it signals inconsistency. Clean solution: go fully reactive (WebFlux only) or call .block() explicitly.
🔗 Kustomize vs Helm for K8s config management▼
Chosen: Kustomize (base + overlays).

Why: Built into kubectl (no extra tools), simpler mental model for beginners — you patch YAML rather than template it. Base + overlays maps cleanly to the dev/prod distinction.

Trade-off: Kustomize has no package management, no versioned chart releases, no dependency graph. For complex multi-team environments with many shared components, Helm's packaging model scales better.

Learning Path Recommendation

1

Start Here — Run the Stack

Run make up, open Grafana at :3000, generate traffic, and trace a request end-to-end. Understand what you're looking at before diving into code.

2

Understand Config Flow

Change a property in config/props/employee/employee.properties, run kubectl rollout restart, and watch the service pick it up without rebuilding the image.

3

Understand JWT Auth

Use Swagger UI at :9000 to get a token. Call the gateway with and without the token. Read JwtAuthenticationFilter.java line by line.

4

Break and Observe

Stop the department service. Watch the employee service's /departments endpoint fail. See the error in Grafana logs. This is how you learn failure modes.

5

Add a Circuit Breaker

Add Resilience4j to the employee service's WebClient call to department. Now the failure returns a fallback response instead of an error. This is the natural next chapter.
