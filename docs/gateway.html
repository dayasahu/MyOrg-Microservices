API Gateway Design Pattern

One door into the system. The gateway routes every request and enforces authentication before anything reaches a business service.

Why an API Gateway?

🚪

Single Entry Point

Clients call one host (api.myorg.com) instead of tracking five service URLs.

🔐

Centralized Auth

JWT validation happens once at the edge — services downstream stay simple.

🧭

Routing in Config

Routes are properties served by the config server, changeable without a rebuild.

🛡️

Topology Hiding

Internal service names & ports never leak to clients.

Routing Rules

Routes are defined in apigateway.properties (served by the config server). Each route matches a path prefix, forwards to a cluster-internal service, and strips the first path segment.

Incoming PathRouted ToFilterService receives
/employee/**http://employee:80StripPrefix=1/api/...
/department/**http://department:80StripPrefix=1/api/...
/auth/**http://auth-service:9000StripPrefix=1/token
 
# apigateway.properties — route 0 (employee)
spring.cloud.gateway.server.webflux.routes[0].id=employee-route
spring.cloud.gateway.server.webflux.routes[0].uri=http://employee:80
spring.cloud.gateway.server.webflux.routes[0].predicates[0]=Path=/employee/**
spring.cloud.gateway.server.webflux.routes[0].filters[0]=StripPrefix=1

JWT Authentication Flow

The gateway implements a reactive GlobalFilter with order -1 (runs before routing). /auth/** is whitelisted so clients can obtain a token; every other route requires a valid Bearer token.

Figure 2 — JWT issuance and gateway validation. The secret is shared between auth-service and gateway via the jwt-secret Kubernetes Secret.

How the Code Solves It

The filter is concise and reactive (non-blocking). Whitelisting, header check, signature verification and a structured log line — all in one place:

@Component
public class JwtAuthenticationFilter implements GlobalFilter, Ordered {

  @Value("${jwt.secret}") private String secret;

  public Mono<Void> filter(ServerWebExchange exchange, GatewayFilterChain chain) {
    String path = exchange.getRequest().getURI().getPath();
    if (path.startsWith("/auth")) return chain.filter(exchange);   // whitelist

    String authHeader = exchange.getRequest().getHeaders().getFirst(AUTHORIZATION);
    if (authHeader == null || !authHeader.startsWith("Bearer ")) {
      exchange.getResponse().setStatusCode(HttpStatus.UNAUTHORIZED);
      return exchange.getResponse().setComplete();           // 401
    }
    try {
      Claims claims = Jwts.parser().verifyWith(signingKey()).build()
          .parseSignedClaims(authHeader.substring(7)).getPayload();
      log.info("JWT validated | subject={} roles={}", claims.getSubject(), claims.get("roles"));
      return chain.filter(exchange);                          // pass through
    } catch (Exception e) {
      exchange.getResponse().setStatusCode(HttpStatus.UNAUTHORIZED);
      return exchange.getResponse().setComplete();
    }
  }
  public int getOrder() { return -1; }   // run before routing
}
Shared secret, two sides. The auth-service signs with jwt.secret; the gateway verifies with the same value. Both read it from the jwt-secret Kubernetes Secret injected as JWT_SECRET. The token carries a 24h expiry (jwt.expiration:86400000 ms).

JWT Secret — Generation & Rotation

Both the auth-service (signing) and the gateway (verifying) read the same HMAC secret from a Kubernetes Secret named jwt-secret, injected as the JWT_SECRET env var.

# auth-service/k8/secret.yaml
apiVersion: v1
kind: Secret
metadata: { name: jwt-secret }
type: Opaque
stringData:
  # DEV ONLY — regenerate before any non-local cluster
  JWT_SECRET: "myDefaultSecretKey1234567890abcdefghij"
# both deployments inject it the same way
env:
  - name: JWT_SECRET
    valueFrom:
      secretKeyRef: { name: jwt-secret, key: JWT_SECRET }

Generate a strong secret

# HMAC-SHA256 needs ≥32 bytes; 48 bytes base64 is plenty
openssl rand -base64 48

Put the value in the Secret (stringData takes plain text — Kubernetes base64-encodes it for you), then re-apply and restart the two consumers:

kubectl apply -f auth-service/k8/secret.yaml -n microservices
kubectl rollout restart deployment/auth-service -n microservices
kubectl rollout restart deployment/apigateway   -n microservices
The default secret is for local dev only. It is committed for convenience — never ship it. For anything beyond a local cluster, generate a fresh secret and source it from a real secret manager (AWS Secrets Manager, Sealed Secrets, Vault) rather than a committed file.
Rotation note. Because the secret is shared and symmetric (HS256), rotating it invalidates all existing tokens. Roll auth-service and the gateway together so signing and verification never drift.
