CI/CD & Argo

The GitHub Actions pipeline that builds, tests, and blue-green deploys each service — and where Argo Rollouts fits versus Argo CD.

The GitHub Actions Pipeline

Each deployable service (employee, department) has a blue-green-deploy.yml workflow that runs on a self-hosted runner on push to master. It is a chain of dependent jobs with a manual approval gate before promotion.

Figure 6 — Nine-stage blue-green pipeline. Any test failure (5–7) triggers automatic rollback to blue.

Stage breakdown

Test Strategies in the Pipeline

The pipeline runs five layers of testing before a single user sees the new version. Each layer catches a different class of problem, and they run in increasing order of cost and realism — the classic testing pyramid applied to a blue-green pipeline.

Figure 7 — The five test layers, cheapest & fastest at the base, most realistic at the top.
LayerStageToolRuns againstWhat it catches
Unit1 · Buildmvn testIndividual classesBusiness-logic bugs in isolation; fastest feedback.
Integration2 · Integrationmvn verify -DskipUnitTestsWired-together Spring contextBeans, config, and component interactions failing together.
Smoke5 · Smokecurl / scripted checksPreview (green) serviceDid the new version actually start & respond? Health + key endpoints return 200.
Performance6 · Perfk6Preview (green) serviceLatency/throughput regressions under synthetic load before going live.
E2E7 · E2ENewman (Postman)Preview (green) serviceFull happy-path API flows across endpoints behave correctly.

Where each runs in the flow

Before the image is built — Unit & Integration

Stages 1–2 fail fast and cheap. If logic or wiring is broken, the pipeline stops before anything is pushed or deployed.

After green is deployed — Smoke, Performance, E2E

Stages 5–7 run against the preview service only — real users are still on blue. These validate the running container, not just the code.

Automated pre-promotion analysis

Independently, Argo Rollouts' AnalysisTemplate hits /actuator/health/readiness 3× on the preview and blocks promotion unless all return UP — a safety net even outside the pipeline.

Any failure → rollback

If smoke, performance or E2E fails, the rollback job (if: failure()) aborts the rollout and undoes it — blue keeps serving, zero user impact.

The honest limit of pre-promotion testing. Stages 5–7 use synthetic traffic and test data on the preview service — they cannot exercise real production load or real user data until traffic actually flips. That is why the real safety net is fast rollback (blue kept 30s) plus post-promotion metric monitoring, not the assumption that the tests caught everything. See Deployment.
Test data lives in the repo. The k6 script (tests/performance/load-test.js) and the Newman collection (tests/e2e/collection.json) are version-controlled alongside the service, so the tests evolve with the code.

Self-Hosted Runner

The pipeline runs on a self-hosted runner because it needs kubectl access to a cluster that isn't publicly reachable. The runner polls GitHub over outbound HTTPS — GitHub never connects inwards — so it works safely behind a firewall.

Build once, deploy many. The image is tagged with the git SHA and that same immutable image is what gets promoted — so what you tested is exactly what goes live. In a multi-env setup, build on a GitHub-hosted runner, then deploy with environment-specific self-hosted runners labelled dev / uat / prod.

Argo Rollouts vs Argo CD

This repo uses Argo Rollouts (progressive delivery). It is often confused with Argo CD (GitOps continuous delivery) — they are different tools from the same family and are complementary.

Argo Rollouts used hereArgo CD not used
PurposeAdvanced deploy strategies (blue-green, canary)Sync Git state → cluster (GitOps)
ReplacesThe K8s Deployment objectThe kubectl apply step

How Argo CD would integrate with GitHub Actions

In a GitOps model the pipeline would not run kubectl. Instead:

CI builds & pushes the image

GitHub Actions builds image:<sha> and pushes it — exactly as today (stages 1–3).

CI updates the manifest in Git

The pipeline bumps the image tag in a Kustomize/Helm file and commits it back to the repo.

Argo CD detects the change

Argo CD, running in the cluster, sees the Git change and syncs it automatically.

Argo Rollouts performs the strategy

Because the manifest is a Rollout, Argo CD hands off to Argo Rollouts for the blue-green/canary flip — the two tools work together.
