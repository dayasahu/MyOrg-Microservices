MyOrg Microservices Platform

A production-grade Spring Boot microservices system on Kubernetes — with an API gateway, centralized configuration, a five-layer test pyramid gating every deployment, full LGTM observability, and zero-downtime blue-green releases driven by GitHub Actions and Argo Rollouts.

Java 21 Spring Boot 3.5.8 Spring Cloud 2025.0.0 Kubernetes JUnit 5 · Mockito k6 Performance Newman / Postman E2E Argo Rollouts Grafana LGTM GitHub Actions

This documentation covers every layer of the platform — from how a JWT travels through the gateway, to how five testing layers gate each deployment, down to how metrics, logs and traces are visualised. Written so anyone cloning the repo can understand the design decisions and stand the system up on Windows with one command.

What's Inside

⊟

API Gateway

Spring Cloud Gateway routes all traffic and validates JWTs before a request reaches any service.

⛁

Config Server

Centralized Spring Cloud Config in native mode, backed by Kubernetes ConfigMaps per environment.

🔐

JWT Auth

Dedicated auth-service issues HS256 tokens; the global gateway filter enforces them platform-wide.

🧪

Five-Layer Testing

Unit → Integration → Smoke → Performance (k6) → E2E (Newman). Every layer must pass before green goes live.

📊

LGTM Observability

Loki, Grafana, Tempo, Prometheus + Grafana Alloy as a single DaemonSet collector.

🚀

Blue-Green + CI/CD

Argo Rollouts with automated analysis, GitHub Actions pipeline, and a manual approval gate before promotion.

  • Kubernetes — orchestration, namespaces, ConfigMaps

End-to-End Architecture

The diagram below shows how a single authenticated request flows through the system, and how every service streams telemetry to the observability stack running in a separate namespace.

Figure 1 — End-to-end request flow and telemetry pipeline across the two namespaces.

A Request, Step by Step

Client hits the ingress

A request to http://api.myorg.com/employee/api/employees resolves (via the hosts file locally) to the NGINX ingress controller.

Ingress forwards to the gateway

The ingress sends everything under / to the apigateway service on port 80.

Gateway validates the JWT

The global JwtAuthenticationFilter checks the Authorization: Bearer header. /auth/** is whitelisted; everything else needs a valid token.

Gateway routes & strips the prefix

/employee/** → http://employee:80 with StripPrefix=1, so the employee service receives /api/employees.

Service-to-service call

The employee service uses a reactive WebClient to call http://department/api/list for the departments endpoint.

Telemetry streams out

Throughout, each service emits traces & metrics over OTLP to Grafana Alloy, while Alloy also tails pod stdout logs — all forwarded to Tempo, Prometheus and Loki.
