The Telemetry Pipeline

Figure 4 — Services emit OTLP + stdout → Alloy fans out to Loki/Tempo/Prometheus → Grafana queries all three.

How each pillar flows

Metrics
Services expose /actuator/prometheus; Prometheus scrapes via K8s DNS, and Alloy also forwards OTLP metrics via remote-write.
Traces
Micrometer Tracing + OTLP exporter → Alloy OTLP receiver (:4318) → Tempo.
Logs
Logback writes JSON to stdout (with traceId/spanId); Alloy tails pod logs from the node and ships to Loki.

How Services Are Wired

# every service's properties point at Alloy (K8s DNS)
management.tracing.sampling.probability=1.0
management.otlp.tracing.endpoint=${OTEL_TRACES_ENDPOINT:http://alloy.observability.svc.cluster.local:4318/v1/traces}
management.otlp.metrics.export.url=${OTEL_METRICS_ENDPOINT:http://alloy.observability.svc.cluster.local:4318/v1/metrics}

# traceId/spanId injected into every log line for correlation
logging.pattern.level=%5p [${spring.application.name:},%X{traceId:-},%X{spanId:-}]

Creating a Dashboard

Open Explore

Grafana → Explore, pick the Prometheus datasource, switch to Code mode.

Write a query

e.g. JVM heap used: jvm_memory_used_bytes{app="employee", area="heap"}

Add to dashboard

Click Add to dashboard → New panel → choose a visualization (time series, stat, table).

Make it reusable

Add a dashboard variable $service = label_values(jvm_memory_used_bytes, app) with multi-select + "All", then use app=~"$service" in queries.

Persist as code

Export JSON → add it to the Grafana grafana-dashboards ConfigMap so it auto-provisions on restart.

Handy PromQL

# request rate per second
sum(rate(http_server_requests_seconds_count{app="employee"}[1m]))
# error rate %
100 * sum(rate(http_server_requests_seconds_count{app="employee",status=~"5.."}[1m]))
    / sum(rate(http_server_requests_seconds_count{app="employee"}[1m]))
# p95 latency (ms)
histogram_quantile(0.95, sum by (le) (rate(http_server_requests_seconds_bucket{app="employee"}[5m]))) * 1000
# heap % of limit
100 * jvm_memory_used_bytes{area="heap"} / jvm_memory_max_bytes{area="heap"}

Monitor, Trace & Correlate a Flow

Generate traffic

Hit /employee/api/departments (employee → department) a few times.

Find a log

Explore → Loki → {namespace="microservices", app="employee"}. Each line carries a traceId.

Jump to the trace

Click the traceId — Grafana opens the matching trace in Tempo, showing the full span tree (gateway → employee → department).

Check the metrics

Open the Spring Boot dashboard to see the request rate spike and the latency for those calls.

Comparison With Other Frameworks

StackLogsMetricsTracesNotes
LGTM + Alloy (this repo)LokiPrometheusTempoOpen-source, one UI, cheap, single agent. Chosen
ELK / ElasticElasticsearchMetricbeatAPMPowerful full-text search; heavier & costlier to run.
Jaeger (traces only)——JaegerGreat tracing UI, but separate tools needed for logs/metrics. Tempo wins for Grafana-native correlation.
Datadog✓✓✓Best DX, but expensive SaaS; teams often migrate off it to LGTM at scale.
OTel Collector (vendor-neutral)✓✓✓CNCF standard; Alloy is the Grafana-native equivalent and was chosen for tighter Grafana integration.
Why Alloy over Promtail / OTel Collector? Promtail is deprecated and logs-only; a standalone OTel Collector is vendor-neutral but needs more wiring. Grafana Alloy is the actively-maintained successor that handles all three signals in one DaemonSet, and it tails pod stdout automatically — so services need no in-process log appender.
