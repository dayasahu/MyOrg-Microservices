Architecture

Config Server

Centralized configuration with Spring Cloud Config in native mode, backed by Kubernetes ConfigMaps generated by Kustomize.

How It Works

Figure 3 — Properties → Kustomize → ConfigMaps → Config Server → services request config by name at startup.

The Config Server runs in native (file-based) mode. Three ConfigMaps are mounted into the pod as read-only volumes, and an env var tells Spring where to look:

# configserver/k8/deployment.yaml (env)
SPRING_PROFILES_ACTIVE: native
SPRING_CLOUD_CONFIG_SERVER_NATIVE_SEARCH_LOCATIONS:
  file:/config/employee,file:/config/apigateway,file:/config/department
SERVER_PORT: 8071
# volumeMounts — each ConfigMap mounted at its own path
- name: employee-config   mountPath: /config/employee   readOnly: true
- name: apigateway-config mountPath: /config/apigateway readOnly: true
- name: department-config mountPath: /config/department readOnly: true

How Services Consume Config

At startup, each service imports its config from the server, keyed by its application name:

# employee rollout.yaml (env)
SPRING_CONFIG_IMPORT: "optional:configserver:http://configserver:8071"
SPRING_APPLICATION_NAME: employee   # → server serves employee.properties

Kustomize Overlays

Config is plain .properties in Git, layered with dev/prod overlays. The configMapGenerator turns them into ConfigMaps:

# config-map-k8/employee/base/kustomization.yaml
configMapGenerator:
  - name: employee-config
    files: [ employee.properties ]
generatorOptions:
  disableNameSuffixHash: true   # stable name so the server volume mount matches

# overlays/dev merges environment-specific values
configMapGenerator:
  - name: employee-config
    behavior: merge
    files: [ employee-dev.properties ]

Why Config Server vs Alternatives

OptionProsConsVerdict
Spring Cloud Config (native)Spring-native, /actuator/refresh, single source for all services, Git-reviewableOne more service to runChosen ✓
Plain ConfigMaps as env varsSimplest, no extra serviceNo Spring refresh, scattered, no central viewToo basic
Config in the JARNothing externalRebuild for every change; same image can't span envsRejected
HashiCorp Vault / ConsulPowerful, secrets + dynamic configHeavy to operate for a training systemOverkill here
Git-backed Config ServerVersioned config in a Git repoNeeds Git access from cluster; native mode is simpler locallyProd option
The payoff: change a route, a port, or an OTLP endpoint by editing a .properties file and re-applying the ConfigMap — no image rebuild, and the same image promotes unchanged across environments.
