Repository Map

A folder-by-folder tour so you know exactly where everything lives the moment you clone the repo.

Clone the Repository

Everything lives in a single monorepo — github.com/dayasahu/MyOrg-Microservices. One clone gives you all five services, the Kubernetes manifests, the Kustomize config, the observability stack, the tooling, and this documentation site.

git clone https://github.com/dayasahu/MyOrg-Microservices.git
cd MyOrg-Microservices

# bring the whole platform up (after prerequisites — see Setup Guide)
make bootstrap   # one-time: NGINX Ingress, Argo Rollouts, K8s Dashboard
make up          # build + deploy observability + services + ingress
Monorepo. All services and orchestration files live together, so the root Makefile and kustomization.yaml resolve every path out of the box — no folder juggling, one command to run the lot.
History preserved. The repo was assembled with git subtree, so each service keeps its original commit history under its subfolder.

Anatomy of a Service

Every service follows the same shape. Using employee/ as the example:

employee/
├── src/main/java/com/daya/emp/employee/
│   ├── EmployeeApplication.java     Spring Boot entry point
│   ├── controller/EmployeeController.java   REST endpoints
│   ├── config/WebClientConfig.java  reactive client → department
│   └── model/Employee.java
├── src/main/resources/
│   └── logback-spring.xml           JSON stdout logging (traceId/spanId)
├── k8/
│   ├── rollout.yaml                 Argo Rollout (blue-green)
│   ├── service.yaml                 active (blue) Service
│   ├── service-preview.yaml         preview (green) Service
│   └── analysis-template.yaml       smoke-test for pre-promotion
├── tests/
│   ├── performance/load-test.js     k6 load test
│   └── e2e/collection.json          Newman/Postman collection
├── .github/workflows/
│   └── blue-green-deploy.yml        CI/CD pipeline
├── Dockerfile                       multi-stage build (Maven → JRE)
├── mvnw.cmd / pom.xml               build (Java 21, Spring Boot 3.5.8)
└── ...
Blue-green services use rollout.yaml, not deployment.yaml. employee and department ship via Argo Rollouts; auth-service, ms-api-gateway and configserver use plain Deployments.

Configuration — config-map-k8/

config-map-k8/
├── employee/
│   ├── base/        employee.properties + kustomization.yaml
│   └── overlays/
│       ├── dev/     employee-dev.properties (merged)
│       └── prod/    configmap.yaml
├── department/      (same shape)
├── apigateway/      (same shape — holds the gateway routes!)
└── kustomization.yaml

These generate the ConfigMaps that the Config Server mounts. The gateway routes live in apigateway/base/apigateway.properties. See Config Server.

Applied as a unit with kubectl apply -k observability/k8 (or make obs-up). See Observability.
