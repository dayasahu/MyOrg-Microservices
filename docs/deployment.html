Deployment Patterns & Blue-Green

How the platform ships changes with zero downtime — and the exact Argo Rollouts configuration that makes it safe.

Two Strategies in One Repo

StrategyUsed byManifestWhy
Rolling Update (K8s default)apigateway, auth-service, configserverdeployment.yamlInternal/infra services; brief overlap acceptable
Blue-Green (Argo Rollouts)employee, departmentrollout.yamlUser-facing APIs; need zero downtime + preview testing
One controller per workload. A service uses either a Deployment or a Rollout — never both, or the two controllers fight over the same pods.

How Blue-Green Works Here

Figure 5 — Blue-green lifecycle: deploy green → analyse on preview → flip active service → keep blue briefly for rollback.

The Rollout Manifest

# employee/k8/rollout.yaml (strategy section)
strategy:
  blueGreen:
    activeService: employee            # live traffic (blue)
    previewService: employee-preview   # new version under test (green)
    autoPromotionEnabled: false       # never auto-flip
    prePromotionAnalysis:
      templates:
        - templateName: smoke-test     # must pass before promotion
      args:
        - name: preview-url
          value: http://employee-preview.microservices.svc.cluster.local:8080
    scaleDownDelaySeconds: 30           # keep blue 30s for instant rollback

The AnalysisTemplate (automated smoke test)

# employee/k8/analysis-template.yaml
kind: AnalysisTemplate
metadata: { name: smoke-test }
spec:
  metrics:
    - name: health-check
      count: 3                       # run 3 times
      interval: 10s                   # 10s apart
      failureLimit: 1
      successCondition: result == "UP"
      provider:
        web:
          url: "{{args.preview-url}}/actuator/health/readiness"
          jsonPath: "{$.status}"

Active vs Preview Services

Argo Rollouts manages the pod-template-hash selector on two services so traffic is routed correctly without you touching selectors:

employee (active)

Points at the stable ReplicaSet — real user traffic.

employee-preview (preview)

Points at the new ReplicaSet — where smoke tests and manual checks run.

Operating a Rollout

# watch the rollout
kubectl argo rollouts get rollout employee -n microservices --watch

# set the new image (kicks off green)
kubectl argo rollouts set image employee employee=dayasahu6077/employee:<sha> -n microservices

# promote green → active (after analysis passes)
kubectl argo rollouts promote employee -n microservices

# instant rollback
kubectl argo rollouts abort employee -n microservices
kubectl argo rollouts undo  employee -n microservices

What Blue-Green Does & Doesn't Guarantee

✅ Guarantees

  • Zero downtime during the switch
  • New version tested on preview before going live
  • Instant rollback (blue kept 30s)

⚠️ Does not guarantee

  • Bug-free release — synthetic tests can't cover every edge case
  • Real production load behaviour (green sees no real traffic pre-promotion)
The real safety net is fast rollback, not perfect testing. For higher confidence, layer canary traffic shifting (also supported by Argo Rollouts) and feature flags on top — promote the code dark, then ramp real users 1% → 100%.
