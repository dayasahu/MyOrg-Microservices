Setup Guide (Windows)

Everything you need to install, and the exact order to bring the platform up from a fresh clone on a Windows machine.

2. Installing Make on Windows

Windows has no built-in make, but the whole workflow is driven by it (make up, make down, …). Pick one of the options below.

Option A — Chocolatey (recommended)

Install Chocolatey (one-time)

Open PowerShell as Administrator and run:

Set-ExecutionPolicy Bypass -Scope Process -Force; `
[System.Net.ServicePointManager]::SecurityProtocol = `
  [System.Net.ServicePointManager]::SecurityProtocol -bor 3072; `
iex ((New-Object System.Net.WebClient).DownloadString('https://community.chocolatey.org/install.ps1'))

Close and reopen the terminal so choco is on your PATH.

Install Make

choco install make -y

Verify

make --version   # should print GNU Make x.x

Option B — winget

# Ships with modern Windows 10/11
winget install GnuWin32.Make
# then add its bin folder to PATH, e.g.:
# C:\Program Files (x86)\GnuWin32\bin

Option C — Scoop

scoop install make
No Make at all? Every target is just a few commands — you can read the Makefile and run them by hand. See the Makefile & Bootstrap page, which lists exactly what each target runs.
Git Bash / WSL note. If you run the workflow from Git Bash or WSL instead of PowerShell, make may already be present (WSL: sudo apt install make). The Makefile targets call PowerShell scripts for bootstrap/health, so running from native Windows PowerShell with Chocolatey's make is the smoothest path.

3. Docker Hub & Registry Login

The build/push targets reference images under the dayasahu6077/* Docker Hub account. To push, you must be logged in to Docker and have write access to that namespace.

You cannot push to someone else's account. make push targets dayasahu6077/.... If you cloned this repo, you do not have write access to that namespace — the push will return 403 denied. Use one of the options below.

Prerequisite — log in to Docker

# Authenticates and stores a token in ~/.docker/config.json
docker login
# username + a Docker Hub access token (Account → Security → New Access Token)

Option A — Use your own Docker Hub account

The image namespace is a single REGISTRY variable (default dayasahu6077). Override it on the command line — no file edits needed:

docker login
make up REGISTRY=your-dockerhub-username
# build/push/deploy all use YOUR namespace

For the CI pipeline, set a repository variable instead: Settings → Secrets and variables → Actions → Variables → New variable → REGISTRY = your-dockerhub-username (plus the DOCKERHUB_USERNAME / DOCKERHUB_TOKEN secrets).

Option B — Local-only, no push needed

With Docker Desktop's built-in Kubernetes, images you build locally are already visible to the cluster — so you can skip make push entirely:

make build        # build images locally
make deploy       # deploy using the local images
# (do NOT run make push)
imagePullPolicy caveat. The manifests use imagePullPolicy: Always, which forces Kubernetes to pull from the registry even when a local image exists. For the local-only path, change it to IfNotPresent on the workloads you're iterating on so the cluster uses your freshly-built local image.

How the CI pipeline authenticates

The GitHub Actions blue-green workflow does not rely on your local login. It logs in using repository secrets, so the runner can push without anyone's interactive credentials:

# in blue-green-deploy.yml
docker login -u ${{ secrets.DOCKERHUB_USERNAME }} \
             -p ${{ secrets.DOCKERHUB_TOKEN }}

Set these under GitHub → repo → Settings → Secrets and variables → Actions: DOCKERHUB_USERNAME and DOCKERHUB_TOKEN (a Docker Hub access token).

4. Hosts File Entries

The platform uses clean ingress hostnames. Map them to localhost. Edit C:\Windows\System32\drivers\etc\hosts as Administrator and add:

127.0.0.1   api.myorg.com
127.0.0.1   grafana.myorg.com
127.0.0.1   prometheus.myorg.com
127.0.0.1   dashboard.myorg.com

5. One-Time Cluster Bootstrap

A brand-new cluster needs three pieces of infrastructure installed once. The scripts/bootstrap.ps1 helper does all three:

🌐

NGINX Ingress

The single load balancer that serves every hostname.

🚀

Argo Rollouts

The controller that powers blue-green deployments.

📺

K8s Dashboard

Web UI for the cluster, exposed on NodePort 30443.

# From the repo root — installs ingress, Argo Rollouts, dashboard
make bootstrap

# or directly:
powershell -ExecutionPolicy Bypass -File scripts/bootstrap.ps1
Run bootstrap only once per cluster. It is idempotent but unnecessary on subsequent runs. make up handles everyday deploys.

6. Bring the Whole Platform Up

A single command builds images, deploys observability, deploys all services, and applies the ingress:

make up

Under the hood this runs, in order:

obs-up

Applies observability/k8 and waits for Loki, Tempo, Prometheus, Grafana (and Alloy) to be ready.

build

Builds Docker images for all five services.

push

Pushes images to Docker Hub (dayasahu6077/*).

deploy

Creates the microservices namespace, applies the root Kustomize, and sets images.

ingress

Applies apigateway-ingress.yaml for api.myorg.com.

Check status & logs

make status   # pods/services in both namespaces
make logs     # tail all service logs
make health   # run the health-check script
make down     # tear everything down

7. Kubernetes Dashboard + Static Token

The bootstrap script exposes the dashboard as a NodePort service and creates an admin-user service account bound to cluster-admin.

Access the dashboard

# Open in browser (accept the self-signed cert warning):
https://localhost:30443
# or via ingress:
https://dashboard.myorg.com

Generate a login token

# Short-lived token (default ~1 hour):
kubectl -n kubernetes-dashboard create token admin-user
Need a long-lived (static) token? Create a Secret of type kubernetes.io/service-account-token bound to the account, then read it back: 
# 1. Create a non-expiring token secret
kubectl apply -f - <<EOF
apiVersion: v1
kind: Secret
metadata:
  name: admin-user-token
  namespace: kubernetes-dashboard
  annotations:
    kubernetes.io/service-account.name: admin-user
type: kubernetes.io/service-account-token
EOF

# 2. Read the static token
kubectl -n kubernetes-dashboard get secret admin-user-token \
  -o jsonpath="{.data.token}" | base64 -d

Paste the resulting token into the dashboard's Token login field.

9. Smoke-Test the Platform

# 1. Get a JWT from the auth-service (through the gateway)
curl -X POST http://api.myorg.com/auth/token \
  -H "Content-Type: application/json" \
  -d "{\"userId\":\"12345\"}"

# 2. Call a protected endpoint with the token
curl http://api.myorg.com/employee/api/employees \
  -H "Authorization: Bearer <token>"

# 3. Service-to-service call (employee → department)
curl http://api.myorg.com/employee/api/departments \
  -H "Authorization: Bearer <token>"
Common gotcha: host.docker.internal does not resolve inside K8s pods. Always use Kubernetes DNS (e.g. alloy.observability.svc.cluster.local) or the real machine IP for pod-to-host communication — never host.docker.internal in pod env vars.
