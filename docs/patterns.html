Architecture

Design Patterns & Rationale

Each pattern below is used deliberately. For every one: what it is, why it was chosen here, and what the alternatives were.

API Gateway Pattern

What: A single entry point (ms-api-gateway) sits in front of all services, handling routing, cross-cutting auth, and prefix stripping.

Why here: Clients need one URL, not five. Auth is enforced once at the edge so downstream services stay simple. Routing rules live in config, not code.

Alternatives considered:

  • Client-side routing — every client must know every service URL. Brittle, leaks topology.
  • Dedicated gateway product (Kong, APISIX) — powerful but heavier; Spring Cloud Gateway integrates natively with the Spring stack.
  • Spring Cloud Gateway ✓ — reactive, code-light, routes defined as properties served by the config server.

Externalized Configuration Pattern

What: Configuration is pulled from a central Config Server at startup, not baked into images.

Why here: The same image promotes unchanged from dev → prod; only the served config differs. Ports, routes and OTLP endpoints change without a rebuild.

Alternatives:

  • Config baked in JAR — requires a rebuild for any change.
  • Pure ConfigMaps as env vars — works, but no Spring refresh, no single source for all services.
  • Spring Cloud Config (native) ✓ — Spring-native, supports /actuator/refresh, backed by ConfigMaps in-cluster.

Token-Based Authentication (JWT)

What: Stateless HS256 JWTs minted by the auth-service, verified at the gateway.

Why here: Stateless means no session store; any gateway replica can validate any token using the shared secret. Downstream services never see auth logic.

Alternatives:

  • Session cookies + shared store — stateful, needs sticky sessions or Redis.
  • OAuth2 / OIDC provider (Keycloak) — the production-grade choice; heavier to run for a training system.
  • Self-issued JWT ✓ — minimal, demonstrates the validation flow clearly.

Single Collector / Agent Pattern

What: Services push telemetry to one endpoint (Grafana Alloy) rather than knowing about Loki, Tempo and Prometheus individually.

Why here: Decouples producers from backends. Swap Tempo for Jaeger and not one service changes — only Alloy's config does.

Alternatives:

  • Each service writes to each backend — tight coupling, repeated config everywhere.
  • Promtail (logs only) — deprecated, and only solves logs.
  • Grafana Alloy DaemonSet ✓ — one agent for logs + traces + metrics, the actively-maintained successor to Promtail and the Grafana Agent.

Blue-Green Deployment Pattern

What: Two versions exist briefly — "blue" (live) and "green" (new). Traffic flips only after the green version passes analysis.

Why here: Zero downtime and instant rollback. The new version is fully tested on a preview service before any user sees it.

Alternatives:

  • Rolling update (default K8s) — simple, but users hit a mix of old + new during rollout, and rollback is slow.
  • Canary — great for gradual traffic shifting; more moving parts. Argo Rollouts supports it too.
  • Blue-Green (Argo Rollouts) ✓ — clean all-or-nothing switch with automated pre-promotion smoke analysis. See Deployment.

Config-as-Volume (ConfigMap mount) Pattern

What: The Config Server reads property files from ConfigMaps mounted as read-only volumes at /config/{service}.

Why here: Kustomize generates the ConfigMaps from plain .properties files with dev/prod overlays — config is reviewable in Git and environment-specific.

Pattern selection principle used throughout: prefer the Spring-native, Kubernetes-native option that keeps services thin and pushes cross-cutting logic to the platform layer — only reaching for heavier external products when scale genuinely demands it.
