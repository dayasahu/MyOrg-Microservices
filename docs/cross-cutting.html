Cross-Cutting Concerns

Every distributed system must solve the same recurring problems. Here is exactly which concern this repo addresses, and the concrete mechanism it uses.

"Cross-cutting concerns" are responsibilities that span all services rather than belonging to any single one — configuration, security, routing, resilience, observability and deployment. Solving them once, consistently, is what separates a pile of services from a platform.

Configuration

Instead of baking config into each image, all runtime properties live in a central place. The Config Server reads .properties files (mounted from ConfigMaps) and serves them to each service at startup based on its SPRING_APPLICATION_NAME.

Why it matters: one place to change a port, a route, or an OTLP endpoint — no rebuild required. See the Config Server page.

Security

Authentication is centralized at the edge. The auth-service mints tokens; the gateway is the single chokepoint that verifies them. Downstream services trust the gateway and stay simple.

// Gateway: JWT validated for everything except /auth/**
if (path.startsWith("/auth")) return chain.filter(exchange);  // whitelist
// else require a valid Bearer token or return 401

Service Discovery

No Eureka or Consul needed — Kubernetes provides DNS-based discovery out of the box. A service named department is reachable cluster-wide at http://department. The employee service relies on exactly this:

// employee → department, by service name
webClient.get().uri("http://department/api/list") ...

Resilience & Health

Each pod exposes Spring Boot Actuator health groups, wired into Kubernetes probes:

Liveness

/actuator/health/liveness — if this fails, K8s restarts the pod.

Readiness

/actuator/health/readiness — if this fails, K8s stops sending it traffic.

Observability

Every service is instrumented with OpenTelemetry. Traces and metrics are pushed over OTLP to Grafana Alloy, which also tails container stdout for logs. The shared traceId in each log line lets you jump from a log straight to its full trace. Full details on the Observability page.
