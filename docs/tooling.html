Makefile & Bootstrap

The glue that spins up the entire environment with a single command, and the one-time scripts that prepare a fresh cluster.

The Makefile

The root Makefile is the everyday control panel. It orchestrates Kustomize, Docker, and the observability stack so you rarely type raw kubectl.

Prerequisite: Windows has no built-in make. Install it first (choco install make) — see Setup Guide → Installing Make on Windows. No Make? Each target is just a handful of commands you can run by hand from the table below.
TargetWhat it does
make bootstrapOne-time: installs NGINX Ingress, Argo Rollouts, K8s Dashboard (calls scripts/bootstrap.ps1).
make upFull stack: obs-up → build → push → deploy → ingress.
make downDeletes the app, ingress, and both namespaces.
make obs-upApplies observability/k8 and waits for all components ready.
make build / pushBuilds / pushes all five service images (IMAGE_TAG configurable).
make deployCreates namespace, kubectl apply -k ., sets images, waits for rollout.
make ingressApplies the API gateway ingress (api.myorg.com).
make statusPods/services/ingress across both namespaces.
make healthRuns scripts/health-check.ps1.
make logsTails logs for all services by the shared label.

What make up chains together

up: obs-up build push deploy ingress

# obs-up — observability into K8s, then wait for readiness
obs-up:
	kubectl apply -k observability/k8
	kubectl rollout status deployment/loki       -n observability --timeout=120s
	kubectl rollout status deployment/grafana    -n observability --timeout=120s
	# ...tempo, prometheus (Deployments) + alloy (DaemonSet)

# deploy — namespace + Kustomize + set images + wait
deploy:
	kubectl create namespace microservices --dry-run=client -o yaml | kubectl apply -f -
	kubectl apply -k .
	# Deployments use the standard plugin...
	kubectl -n microservices set image deployment/apigateway apigateway=dayasahu6077/ms-api-gateway:$(IMAGE_TAG)
	# ...but employee & department are Argo Rollouts, so:
	kubectl argo rollouts -n microservices set image employee employee=dayasahu6077/employee:$(IMAGE_TAG)
	# then wait: kubectl argo rollouts status employee
Deployments vs Rollouts. employee and department are Argo Rollout objects, not Deployments — so they use kubectl argo rollouts set image / status, while configserver, apigateway and auth-service use the standard kubectl set image / rollout status deployment/…. The Makefile handles both correctly.
IMAGE_TAG is configurable. Default is latest; override per build: make deploy IMAGE_TAG=v1.2.0. In CI this is set to the git SHA for immutable images.

Root Kustomization

make deploy's kubectl apply -k . uses the root kustomization.yaml as the single source of truth for the microservices namespace. It applies resources in dependency order:

ConfigMaps first

employee / department / apigateway dev overlays — must exist before the config server mounts them.

Shared secret

jwt-secret for gateway + auth-service.

Config server

Deployment + service, so other services can pull config.

Business services

employee & department as Rollouts (+ preview service & analysis template); gateway as a Deployment.

Auth service

Deployment + service.

A shared label is stamped on everything for easy selection:

labels:
  - pairs:
      app.kubernetes.io/part-of: microservices-training
      environment: dev
    includeSelectors: true

scripts/bootstrap.ps1

Run once on a new cluster. It installs the three infrastructure pieces the platform assumes exist:

StepInstallsDetail
1NGINX Ingress ControllerThe single LB serving every hostname; waits for the controller pod to be ready.
2Argo RolloutsCreates argo-rollouts namespace and applies the controller; powers blue-green.
3Kubernetes DashboardInstalls the dashboard, exposes it as NodePort 30443, and creates an admin-user + cluster-admin binding.

Flags let you skip pieces already installed:

powershell -ExecutionPolicy Bypass -File scripts/bootstrap.ps1 -SkipIngress -SkipArgo

Other Helper Files

scripts/bootstrap.ps1

One-time cluster setup — installs NGINX Ingress, Argo Rollouts, and the K8s Dashboard. Invoked by make bootstrap.

scripts/health-check.ps1

Invoked by make health — verifies pod readiness and that key services have endpoints.

Dockerfile (per service)

Multi-stage build: Maven builds the jar, then a slim eclipse-temurin:21-jre runtime — no JDK needed on the host.

JWT secret

Generated with openssl rand -base64 48 and stored in the jwt-secret Secret — see API Gateway → JWT Secret.

Typical Daily Loop

make bootstrap          # once per new cluster
make up                 # build + deploy everything
make status             # confirm pods are Running
# ... develop, change code ...
make build push deploy  # ship a new image
make logs               # watch output
make down               # tear it all down
One-command environment. The combination of the Makefile (orchestration), bootstrap script (one-time infra), root Kustomization (what to deploy) and multi-stage Dockerfiles (how to build) means anyone can clone the repo and reach a running platform with make bootstrap then make up.
